# Вопрос

Расскажи подробнее про синтаксис исключений в /etc/dnf/dnf.conf Как определяются исключения через этот файл. После этого опиши в чем отличия и ограничения в сравнении с python3-dnf-plugin-versionlock

# Рассуждения

Мы рассмотрим два механизма управления обновлениями пакетов в DNF: через параметр exclude в /etc/dnf/dnf.conf и через плагин versionlock. Сравним синтаксис, поведение, гибкость и ограничения.

1. Синтаксис исключений через /etc/dnf/dnf.conf
Файл /etc/dnf/dnf.conf — основной конфигурационный файл DNF. В секции [main] можно задать параметр exclude, который принимает список имён пакетов или glob-шаблонов, разделённых пробелами или запятыми. Эти пакеты будут полностью игнорироваться при любой операции DNF: install, update, upgrade, distro-sync (но не remove, если пакет уже установлен, и не reinstall).

Пример содержимого /etc/dnf/dnf.conf:
~~~
ini
[main]
gpgcheck=1
installonly_limit=3
exclude=git-lfs* kernel-5.14.0-* httpd-2.4.*
~~~
Здесь:

- git-lfs* — исключает любые пакеты, имя которых начинается с git-lfs (например, git-lfs, git-lfs-doc), любых версий и архитектур.

- kernel-5.14.0-* — исключает все пакеты ядра с версией, начинающейся на 5.14.0- (образы ядра, модули).

- httpd-2.4.* — исключает пакет httpd версии 2.4.x (точный шаблон).

Правила синтаксиса:
1. Разделители: Пробелы или запятые.

2. Glob-символы:

- \* — любое количество любых символов.

- ? — один любой символ.

- [...] — любой символ из указанного диапазона.

3. Регистрозависимость? DNF работает с именами пакетов в нижнем регистре, так что обычно регистр не важен.

4. Можно группировать с помощью кавычек, если имена содержат пробелы (редко).

Проверка активных исключений:
~~~
bash
dnf config-manager --dump | grep exclude
# или
grep ^exclude /etc/dnf/dnf.conf
~~~
Поведение:
- При dnf update все пакеты, подпадающие под шаблоны исключений, будут пропущены (не обновлены, не установлены).

- При явной установке dnf install git-lfs — если пакет исключён, DNF выдаст ошибку: No match for argument: git-lfs.

- Если пакет уже установлен, dnf update не будет его обновлять (останется текущая версия).

- Исключения не защищают пакет от удаления (например, если он входит в группу, удаляемую при dnf group remove или при dnf autoremove).
---
2. Механизм python3-dnf-plugin-versionlock (версионные блокировки)
Плагин versionlock добавляет возможность зафиксировать конкретные версии пакетов (или маски версий), чтобы DNF не обновлял их ни при каких условиях, но при этом не запрещает установку других версий, если они явно запрошены.

Файл конфигурации: /etc/dnf/plugins/versionlock.list
Каждая строка — это спецификатор пакета в формате, совместимом с dnf versionlock:
~~~
text
# комментарии
git-lfs-2:3.2.0-2.el8.*
httpd-0:2.4.37-62.module+el8.10.0+12345.*
docker-ce-3:24.0.7-1.el8.*
~~~
Синтаксис строки:
~~~
text
[epoch:]name-version-release[.*]
~~~
- epoch (опционально) — число, по умолчанию 0 (не пишется, но может быть указано).

- name — имя пакета.

- version — версия (например, 3.2.0).

- release — релизная часть (например, 2.el8).

- .* — суффикс, означающий «любая архитектура» (поскольку versionlock хранит информацию с учётом архитектуры, .* действует как wildcard для неё).

Примеры из официальной документации:
~~~
text
kernel-0:5.14.0-427.el9.x86_64
nginx-1:1.20.1-10.el9.*
~~~
Управление блокировками:
~~~
bash
# Добавить блокировку для текущей установленной версии пакета
dnf versionlock add git-lfs

# Добавить блокировку с явной версией
dnf versionlock add git-lfs-3.2.0

# Показать все блокировки
dnf versionlock list

# Удалить блокировку для пакета
dnf versionlock delete git-lfs

# Очистить все блокировки
dnf versionlock clear
~~~
Поведение:
- При dnf update заблокированные пакеты не обновляются (строгая фиксация версии).

- При попытке установить другую версию заблокированного пакета (dnf install git-lfs-3.3.0) — DNF выдаст ошибку из-за конфликта с блокировкой.

- Заблокированный пакет можно удалить (dnf remove git-lfs), но после удаления блокировка остаётся в файле (её нужно удалять отдельно).

- Блокировка не мешает установке пакета, если его нет в системе — при установке будет принудительно взята именно та версия, которая указана в блокировке (если она доступна в репозиториях).

3. Сравнение и ограничения
|Характеристика|	exclude в dnf.conf	|versionlock плагин|
|--------------|----------------------|------------------|
| **Гибкость**	|Только глобальный список имён/шаблонов (без указания конкретной версии)	|Можно фиксировать конкретные версии, включая epoch и release|
|**Точность**	|Исключает все версии пакета, удовлетворяющие шаблону	|Точно указывает, какая версия должна остаться|
| **Возможность обновления внутри фиксации**|	Нет (пакет никогда не обновится)	|Можно разрешить обновления в рамках маски: git-lfs-3.2.* — обновятся только минорные версии 3.2.x|
| **Влияние на установку новых версий**|	Запрещает установку любого пакета, подпадающего под шаблон	|Запрещает установку любой версии, отличной от зафиксированной|
| **Совместимость с dnf distro-sync**|	Исключённые пакеты не синхронизируются	|Заблокированные версии сохраняются; если блокировка не совпадает с версией из репозитория, distro-sync предложит понижение (если разрешено) или выдаст ошибку|
| **Простота использования**	|Править один файл вручную	|Требует установки плагина и работы с командами dnf versionlock|
| **Атомарность изменений**	|Изменения вступают в силу сразу после сохранения файла	|Изменения вступают после вызова dnf versionlock add/delete (или редактирования /etc/dnf/plugins/versionlock.list)|
| **Работа с модулями (module streams)**|	Не поддерживает фиксацию модулей	|Поддерживает (через полное имя пакета с модулем)|
| **Производительность**|	Не добавляет накладных расходов	|Плагин перехватывает запросы и сверяет списки, но влияние незаметно|
| **Ограничения**|	Нельзя исключить конкретную версию (только имя или маску). Можно случайно исключить слишком много пакетов.	|При обновлении Python или самого плагина блокировки могут потеряться (надо переустанавливать).|

Ключевые ограничения exclude:
- Нет контроля версии: вы не можете сказать «оставь git-lfs на версии 3.2.0, но разреши обновлять до 3.2.1». Вы только запрещаете весь пакет git-lfs*. Это вынуждает либо полностью блокировать обновления пакета, либо не блокировать вовсе.

- Глобальность: исключение действует на все операции, даже на те, где вы явно хотите установить исключённый пакет. Например, для временной установки новой версии потребуется временно убрать строку из dnf.conf — это неудобно.

- Нет различия между установленными и неустановленными пакетами: если вы исключили git-lfs*, то даже если пакет не установлен, DNF откажется его устанавливать. В versionlock блокировка активна только для обновлений, но не мешает установке пакета, если его нет (хотя при установке будет взята заблокированная версия).

Ключевые ограничения versionlock:
- Зависимость от плагина: если плагин сломан, удалён или обновление Python разрушило его, блокировки перестают работать. Это требует поддержки.

- Не работает для групп пакетов: нельзя заблокировать группу пакетов одной строкой (только перечислять каждый).

- Специфичность архитектуры: по умолчанию блокировка привязывается к архитектуре. Чтобы сделать универсальной, используют суффикс .* (например, git-lfs-3.2.0-2.el8.*). В exclude архитектура игнорируется.

- Запись только основных пакетов: плагин не может заблокировать зависимости или со-пакеты (не нужно, блокировка основного пакета автоматически удерживает его зависимости на тех же версиях, но если зависимость общая для нескольких пакетов — возможны конфликты).
---
4. Вывод: что и когда использовать?
exclude подходит для:
- Быстрых временных блокировок, например, чтобы предотвратить обновление ядра во время важного теста.

- Запрета целых семейств пакетов, которые не должны присутствовать в системе никогда (например, nano*, если политика требует только vim).

- Ситуаций, когда не нужно различать версии, достаточно запретить пакет любыми способами.

versionlock (плагин) подходит для:
- Производственных сред, где требуется точный контроль версий критических пакетов (git-lfs, docker, kubelet).

- Поддержки mixed-версионной среды, когда некоторые хосты должны остаться на старых версиях из-за совместимости с пользовательским ПО.

- Гибкого управления обновлениями в рамках мажорных версий (например, разрешить обновления для git-lfs-3.2.*, но не переходить на 3.3).

- Сценариев, где блокировка должна быть объявлена декларативно и управляема через Ansible.

Рекомендация для вашего проекта:
Используйте versionlock, так как вы уже добавили роль по его установке и настройке. Он даёт необходимую точность и управляемость. exclude можно рассматривать как fallback или для простых случаев, но в рамках вашего playbook проще использовать стандартный плагин DNF.

Если вы хотите дополнительно подстраховаться на случай сбоя плагина, можете добавить в роль lock_versions проверку, что после установки плагина механизм работает (например, выполнить dnf versionlock list и проверить код возврата), и при ошибке — выставить exclude как временную меру. Но это усложнит логику, обычно не требуется.

# Ответ

## Подробный разбор двух механизмов: `exclude` в `dnf.conf` против `versionlock`

### 1. Синтаксис и поведение `exclude` в `/etc/dnf/dnf.conf`

Файл `dnf.conf` — основной конфиг DNF. Параметр `exclude` в секции `[main]` задаёт список имён пакетов (с возможными **глоб-шаблонами** `*`, `?`, `[]`), которые DNF будет **полностью игнорировать** при любых операциях установки, обновления, синхронизации.

**Пример файла `/etc/dnf/dnf.conf`:**
```ini
[main]
gpgcheck=1
exclude=git-lfs* kernel-5.14.0-* nginx-1.20.*
```

**Правила:**
- Разделитель — пробелы или запятые.
- Шаблон применяется к имени пакета целиком (регистр не важен).
- Исключения действуют **глобально** для всех репозиториев.
- Если пакет уже установлен, он не будет обновлён (останется текущая версия).
- Если пакет не установлен, его невозможно будет установить (`dnf install git-lfs` ? ошибка `No match`).

**Проверка:**
```bash
grep ^exclude /etc/dnf/dnf.conf
```

**Ограничения:**
- Нельзя указать конкретную версию или релиз — только имя или маску имени.
- При изменении шаблона легко случайно захватить лишние пакеты (например, `git*` заблокирует `git`, `git-lfs`, `git-core` и т.д.).
- Исключение действует даже при явном указании полного NEVRA (имя-эпоха-версия-релиз-архитектура) — DNF всё равно откажется.
- Нет возможности разрешить обновления в рамках минорной версии (например, `3.2.x` разрешены, а `3.3.x` нет).

---

### 2. Механизм `python3-dnf-plugin-versionlock`

Плагин добавляет команду `dnf versionlock` и хранит список фиксаций в `/etc/dnf/plugins/versionlock.list`.

**Синтаксис строки блокировки:**
```
[epoch:]name-version-release[.*]
```
- `.*` в конце означает «любая архитектура» (подстановка для NEVRA).
- Можно указывать маску в релизной части (например, `git-lfs-3.2.0-2.el8.*`).

**Примеры:**
```
git-lfs-0:3.2.0-2.el8.*
kernel-0:5.14.0-427.el9.x86_64
```

**Управление:**
```bash
# Добавить текущую установленную версию
dnf versionlock add git-lfs

# Добавить конкретную версию
dnf versionlock add git-lfs-3.2.0-2.el8

# Очистить все блокировки
dnf versionlock clear

# Посмотреть список
dnf versionlock list
```

**Поведение:**
- При `dnf update` пакет не обновляется, если его версия отличается от зафиксированной.
- При попытке установить другую версию — ошибка (конфликт с блокировкой).
- Если пакет не установлен, DNF разрешит его установить, но **только версию, совпадающую с блокировкой** (если она доступна в репозиториях).
- Возможно разрешить обновления в рамках маски, например `git-lfs-3.2.*` — тогда любые пакеты `git-lfs-3.2.X` разрешены, а более старшие или младшие — нет.

**Ограничения:**
- Требует установки отдельного пакета и его работоспособности (зависит от версии Python).
- Не работает для групп пакетов (нужно перечислять каждый).
- При обновлении самого DNF или Python может потребоваться переустановка плагина.
- Блокировка сохраняется даже после удаления пакета (её нужно чистить отдельно).

---

### 3. Ключевые отличия (таблица)

| Параметр | `exclude` в `dnf.conf` | `versionlock` плагин |
|----------|------------------------|----------------------|
| **Точность контроля** | Только по имени/маске имени | По полному NEVRA (или маске версии) |
| **Возможность указать версию** | Нет (запрещает любую версию) | Да (фиксирует конкретную версию или диапазон масок) |
| **Обновление внутри мажорной версии** | Нет (полная блокировка) | Да, через маску в версии: `git-lfs-3.2.*` |
| **Установка пакета, если его нет** | Запрещена | Разрешена, но только зафиксированной версии |
| **Зависимость от внешних компонентов** | Нет (читается нативным кодом DNF) | Требует установленного плагина и корректной работы Python |
| **Сброс блокировки** | Удалить строку из файла и, возможно, перезапустить DNF | `dnf versionlock delete <pkg>` или очистка файла |
| **Влияние на `dnf distro-sync`** | Исключённые пакеты не синхронизируются | Заблокированные пакеты остаются на своих версиях (может вызвать ошибку, если версия не совпадает с репозиторием) |
| **Простота автоматизации Ansible** | Модуль `ini_file` или `lineinfile` | Прямые команды `dnf versionlock` (модуль `command`) или редактирование файла списка |

---

### 4. Что произойдёт, если совместить оба метода?

Если пакет одновременно присутствует в `exclude` **и** в `versionlock`, то:
- Исключение имеет **более высокий приоритет** (DNF даже не будет обрабатывать плагин для этого пакета, так как исключение проверяется на раннем этапе).
- Пакет будет полностью невидим для DNF — нельзя ни установить, ни обновить, ни удалить (кроме прямого `rpm -e`).
- **Не рекомендуется** смешивать оба подхода для одних и тех же пакетов, это приводит к путанице.

---

### 5. Рекомендации по выбору для вашего проекта

**Для роли `lock_versions` вы уже выбрали `versionlock` — это правильное решение, потому что:**

- Вы хотите зафиксировать **конкретную версию** `git-lfs`, а не запретить пакет целиком.
- Возможно, позже потребуется разрешить обновления в рамках патчей (например, разрешить `git-lfs-3.2.1`, но не `3.3.0`) — `versionlock` с маской `git-lfs-3.2.*` легко это позволит.
- Вы управляете блокировками декларативно через Ansible, а `dnf versionlock add` идемпотентен (повторный вызов не создаст дублей).

**`exclude` можно использовать только как резервный механизм**, если плагин по какой-то причине сломался, но в production-сценарии лучше ремонтировать плагин, а не переключаться на `exclude`.

**Дополнительная защита в роли `lock_versions`:**

Добавьте проверку, что после установки плагина он действительно работает:
```yaml
- name: Проверить работоспособность плагина versionlock
  ansible.builtin.command:
    cmd: dnf versionlock list
  register: versionlock_test
  failed_when: versionlock_test.rc != 0
```

Если эта проверка падает, playbook аварийно остановится, не начиная обновление системы. Это лучше, чем обновлять систему без активных блокировок.

---

### Заключение

- **`exclude`** — грубый топор для полного запрета пакетов с маскировкой имени.
- **`versionlock`** — скальпель для точной фиксации версий с сохранением возможности установки и контролируемых обновлений.

В вашем сценарии (блокировка `git-lfs` на конкретной версии перед глобальным `dnf update`) правильный выбор — `versionlock`. Он уже реализован в роли, и это соответствует лучшим практикам управления пакетами в RHEL-совместимых дистрибутивах.
