как поведет себя плейбук в таком случае?

# Ответ

Отличный вопрос — он касается **надёжности сценария в условиях реальных сбоев**. Разберём оба случая детально.

## 1. Что делает `ansible.builtin.dnf` с `update_cache: yes`?

Параметр `update_cache: yes` — аналог команды `dnf makecache`. Он **обновляет локальный кэш метаданных** из всех включённых репозиториев.  
**Но** он **не выполняет `dnf clean all`**, то есть старые данные не удаляются. Если репозиторий недоступен:

- DNF попытается скачать свежие метаданные.
- При ошибке соединения/HTTP 404/403 он вернёт **ненулевой код ошибки**.
- Модуль `ansible.builtin.dnf` зафейлит задачу с сообщением вроде `Could not retrieve mirrorlist ...`.

### Поведение плейбука (в текущей реализации из прошлого ответа)

Согласно нашему плейбуку с `rescue`:

```yaml
rescue:
  - name: Запомнить статус ошибки
    set_fact: update_status: {...}
  - name: Продолжить выполнение для остальных хостов
    meta: clear_host_errors
```

То есть:
1. Задача `dnf` падает ? управление уходит в `rescue`.
2. В `rescue` мы записываем статус «Ошибка» для этого хоста.
3. `clear_host_errors` снимает отметку о провале с текущего хоста, позволяя плейбуку **перейти к следующему хосту в группе** (благодаря `serial: 1`).

**Результат:** Группа продолжает обновление на других ВМ, даже если первый сервер не смог обновиться из-за репозитория. При этом в итоговой таблице у первого хоста будет «Ошибка», а остальные — успешны.

### Нужно ли так делать?

**Нет, в production это опасно.** Если репозиторий недоступен, то **все** серверы кластера не смогут корректно обновиться. Обновление части узлов, а части нет, приведёт к **разнородной среде** (разные версии пакетов, ядер), что для Kafka/Logstash кластера недопустимо.

**Правильное поведение:**  
При ошибке на **любом** хосте в группе — **прекратить обновление всей группы** и выдать чёткое сообщение.

## 2. Что если хост зависнет при перезагрузке?

Модуль `ansible.builtin.reboot` работает так:
- Отправляет команду перезагрузки.
- Начинает ждать, пока хост станет недоступным по SSH.
- Затем ждёт, пока хост снова станет доступным (проверка TCP/SSH).
- Если за время `reboot_timeout` (по умолчанию 600 сек) доступ не восстановился — задача падает с ошибкой.

В текущей реализации ошибка перезагрузки также приводит к `rescue` и переходу к следующему хосту через `clear_host_errors`.

**Это катастрофично:**  
- Первый хост завис на этапе загрузки (например, kernel panic, повреждённый initrd).  
- Он никогда не вернётся в строй.  
- Плейбук посчитает это «ошибкой на одном хосте», но продолжит обновлять второй хост.  
- В итоге кластер теряет один узел навсегда, а второй получает новое ядро — состояние становится неконтролируемым.

## Улучшенный подход: прерывание группы при любой критической ошибке

Ниже — как изменить роль `update_services`, чтобы при ошибках `dnf` или `reboot` **прерывать выполнение для всей группы** и не трогать остальные хосты.

### Файл `roles/update_services/tasks/main.yml` (исправленная версия)

```yaml
---
- name: Проверить доступность всех репозиториев и обновить кэш
  block:
    - name: Очистить кэш DNF (избегаем битых метаданных)
      ansible.builtin.command: dnf clean all
      changed_when: false

    - name: Обновить кэш репозиториев
      ansible.builtin.dnf:
        update_cache: yes
  rescue:
    - name: Репозитории недоступны — прерываем обновление всей группы
      ansible.builtin.include_tasks: fail_group.yml
      vars:
        error_message: "Не удалось обновить кэш DNF (репозитории недоступны)"

- name: Проверить, что сервисы ДО обновления запущены
  ansible.builtin.systemd:
    name: "{{ item }}"
    state: started
  loop: "{{ service_names }}"
  register: pre_check
  failed_when: pre_check.results | selectattr('status.ActiveState', 'ne', 'active') | list | length > 0

- name: Остановить сервисы
  ansible.builtin.systemd:
    name: "{{ item }}"
    state: stopped
  loop: "{{ service_names }}"

- name: Обновить пакеты
  ansible.builtin.dnf:
    name: '*'
    state: latest
  register: dnf_result
  rescue:
    - name: Ошибка DNF — прерываем группу
      ansible.builtin.include_tasks: fail_group.yml
      vars:
        error_message: "dnf update завершился ошибкой: {{ dnf_result.msg | default('Unknown error') }}"

- name: Перезагрузить сервер
  ansible.builtin.reboot:
    reboot_timeout: 900          # увеличил до 15 минут
    pre_reboot_delay: 5
    post_reboot_delay: 30
  register: reboot_result
  rescue:
    - name: Сервер не вернулся после перезагрузки — прерываем группу
      ansible.builtin.include_tasks: fail_group.yml
      vars:
        error_message: "Хост не загрузился в течение {{ reboot_result.reboot_timeout | default(900) }} секунд"

- name: Собрать факты после перезагрузки
  ansible.builtin.setup:

- name: Проверить, что сервисы запустились
  ansible.builtin.systemd:
    name: "{{ item }}"
    state: started
  loop: "{{ service_names }}"
  register: post_check
  until: post_check.results | selectattr('status.ActiveState', 'eq', 'active') | list | length == service_names | length
  retries: 6
  delay: 10
  rescue:
    - name: Сервисы не запустились — прерываем группу
      ansible.builtin.include_tasks: fail_group.yml
      vars:
        error_message: "После перезагрузки сервисы {{ service_names | join(', ') }} не активны"

- name: Записать успешный результат
  ansible.builtin.include_tasks: write_result.yml
  vars:
    result_status: "Успешно"
    error_detail: ""
```

### Файл `roles/update_services/tasks/fail_group.yml` (новый)

```yaml
---
- name: Записать ошибку в результат для текущего хоста
  ansible.builtin.include_tasks: write_result.yml
  vars:
    result_status: "Ошибка: {{ error_message }}"
    error_detail: "{{ error_message }}"

- name: Остановить выполнение для всей группы
  ansible.builtin.meta: end_play
```

### Файл `roles/update_services/tasks/write_result.yml` (дополненный)

```yaml
---
- name: Собрать информацию о результатах
  set_fact:
    result_entry:
      host: "{{ inventory_hostname }}"
      group: "{{ group_name }}"
      status: "{{ result_status }}"
      kernel: "{{ ansible_kernel | default('N/A') }}"
      os_version: "{{ ansible_distribution | default('N/A') }} {{ ansible_distribution_version | default('') }}"
      error: "{{ error_detail | default('') }}"

- name: Добавить запись в JSON (атомарно с помощью jq)
  ansible.builtin.shell:
    cmd: |
      tmp=$(mktemp)
      jq --argjson new "{{ result_entry | to_json }}" '. + [$new]' {{ playbook_dir }}/results.json > "$tmp" && mv "$tmp" {{ playbook_dir }}/results.json
    executable: /bin/bash
  delegate_to: localhost
  changed_when: false
```

## Ответы на прямые вопросы

### Как поведет себя `ansible.builtin.dnf` с `update_cache: yes` и `state: latest`, если один из репозиториев недоступен?

**Ответ:** Задача упадёт с ошибкой. Текст ошибки будет содержать проблемный репозиторий и причину (например, `Failed to download metadata for repo 'epel'`).

### Как в текущей версии плейбука (без доработок) поведет себя playbook?

- Выполнится `rescue` ? ошибка запишется для текущего хоста.
- `meta: clear_host_errors` позволит перейти к следующему хосту.
- В итоговой таблице у одного хоста ошибка, остальные продолжают обновление.
- **Это плохо.** Кластер получает неоднородное состояние.

### Что произойдет с плейбуком, если при перезагрузке хост зависнет и не станет доступен?

- Модуль `reboot` будет ждать до `reboot_timeout` (600 сек по умолчанию).
- По истечении таймаута задача упадёт с ошибкой (`Timeout waiting for reboot`).
- В текущей реализации — переход в `rescue`, затем `clear_host_errors` и переход к следующему хосту.
- **Катастрофа:** кластер теряет один узел, а второй обновляется.

В улучшенной версии (с `fail_group.yml`) при любой ошибке (репозиторий, dnf, reboot, сервис) мы вызываем `meta: end_play`, который **немедленно прерывает выполнение для всей текущей группы** (lgs или kaf), не трогая последующие хосты. Это сохраняет кластер в том состоянии, в котором он находился до обновления (один хост, возможно, уже частично обновлён, но второй не тронут). Дальнейшие действия должен предпринять администратор вручную.

## Дополнительные рекомендации

1. **Статус «частичного обновления»**  
   Если первый хост уже перезагрузился, а второй нет (из-за ошибки во время обновления второго) — кластер уже в разнородном состоянии. В таком случае правильно полностью остановить playbook и потребовать ручного вмешательства. Наша конструкция с `end_play` это обеспечивает.

2. **Предварительная проверка репозиториев**  
   Рекомендую вынести проверку доступности репозиториев в отдельный play перед началом обновления группы (с `delegate_to: "{{ item }}"` и модулем `uri` или `command: dnf repolist`). Это даст `fast fail` до остановки сервисов.

3. **Настройка таймаутов reboot**  
   Увеличьте `reboot_timeout` до 900–1200 секунд, особенно если серверы на медленных дисках или с BIOS-паролями.

4. **Логирование**  
   В `write_result.yml` добавьте время выполнения и лог ошибки. Это поможет при разборе инцидентов.

5. **Итоговая таблица**  
   В роли `report` выводите не только таблицу, но и **код возврата** (exit code) Ansible. Если были ошибки — playbook должен завершиться с `rc != 0`. В примере роли `report` уже добавлен блок `fail`.

Итог: предложенный выше улучшенный код делает обновление **безопасным**, останавливая всю группу при любой фатальной ошибке. Это правильный подход для production-кластеров.
