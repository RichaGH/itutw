# Вопрос
Иногда, при обновлении необходимо фиксировать версии пакетов для совместимости пользовательского ПО и пакетной базы хоста. Давай добавим роль, которая проверяет, что для группы (пусть в нашем примере это будет группа lgs) на хосте добавлены в исключения при обновлении или зафиксированы версии пакетов например git-lfs. Если исключения не добавлены, то добавит их. И только потом перейдет к проверкам которые мы уже реализовали ранее. Расскажи сначала как это делается не ansible  скриптом при подключении администратора к хосту по ssh. Затем переходи к реализации на ansible.

# Рассуждения
Мы обсуждаем добавление роли для фиксации версий пакетов (например, git-lfs) в группе lgs. Нужно сначала объяснить ручной процесс через ssh, затем реализовать на Ansible.

Отвечу структурированно: как администратор вручную через SSH фиксирует версию пакета (например, git-lfs) с помощью dnf versionlock или добавлением исключения в конфиг, затем реализация в Ansible в виде отдельной роли, которая проверяет и добавляет фиксацию перед основным обновлением.

Учту, что пакет git-lfs может быть не установлен, но исключение нужно добавить "на будущее". Используем python3-dnf-plugin-versionlock или модуль dnf versionlock add

# Ответ

## Фиксация версий пакетов (например, git-lfs) перед обновлением

### Часть 1. Ручное администрирование через SSH

Допустим, вам нужно «заморозить» версию пакета `git-lfs` на всех хостах группы `lgs`, чтобы при обновлении системы (`dnf update`) этот пакет не обновился, даже если выйдет новая версия. Это называется **версионным локом** (version lock). В RHEL-подобных дистрибутивах (CentOS, RHEL, Fedora, Rocky, Alma) для этого используется плагин DNF `versionlock`.

#### Шаги вручную (на одном хосте):

1. **Установить плагин** `python3-dnf-plugin-versionlock` (если не установлен):
   ```bash
   sudo dnf install -y python3-dnf-plugin-versionlock
   ```

2. **Проверить текущие заблокированные пакеты**:
   ```bash
   sudo dnf versionlock list
   ```

3. **Добавить пакет с конкретной версией** (например, текущую установленную):
   ```bash
   # Сначала узнаем точную версию установленного пакета
   rpm -q git-lfs
   # вывод: git-lfs-3.2.0-2.el8.x86_64

   # Добавляем в лок
   sudo dnf versionlock add git-lfs-3.2.0-2.el8.x86_64
   ```

4. **Проверить, что версия зафиксирована**:
   ```bash
   sudo dnf versionlock list
   # Должна появиться строка: git-lfs-0:3.2.0-2.el8.*
   ```

5. **Удалить фиксацию** (если позже понадобится обновить):
   ```bash
   sudo dnf versionlock delete git-lfs
   # или очистить все: sudo dnf versionlock clear
   ```

**Альтернативный способ через глобальный exclude в `/etc/dnf/dnf.conf`**:
```ini
[main]
exclude=git-lfs*
```
Но этот метод менее гибкий (исключает любой пакет с таким именем на все версии). Используется редко, если нужно заморозить именно одну версию.

**Почему это важно перед массовым обновлением?**
- Пользовательское ПО может зависеть от конкретной версии `git-lfs`. Неожиданное обновление может сломать интеграцию.
- Плагин `versionlock` гарантирует, что даже при `dnf update` пакет останется на указанной версии.

### Часть 2. Реализация на Ansible

Создадим роль `lock_versions`, которая:
- Устанавливает плагин `versionlock`, если его нет.
- Проверяет, зафиксирован ли заданный пакет (например, `git-lfs`) с конкретной версией (или маской).
- Если нет — добавляет фиксацию.
- Если пакет вообще не установлен — можно либо пропустить, либо зафиксировать нужную версию, чтобы при будущей установке не взялась новейшая (зависит от политики).

Роль будет вызываться **перед** основной ролью `update_services` для группы `lgs`.

#### Структура обновлённого `site.yml` (фрагмент):

```yaml
# Play 1: Фиксация версий для группы lgs (ДО обновления)
- name: Зафиксировать версии пакетов на хостах lgs
  hosts: lgs
  gather_facts: yes
  roles:
    - role: lock_versions
      vars:
        packages_to_lock:
          - name: git-lfs
            version: "3.2.0-2.el8"   # мажор.минор-релиз.арх
            # можно использовать маску, например "3.2.*"

# Затем проверка репозиториев и обновление...
```

#### Роль `lock_versions/tasks/main.yml`

```yaml
---
- name: Установить плагин versionlock для DNF
  ansible.builtin.dnf:
    name: python3-dnf-plugin-versionlock
    state: present

- name: Заблокировать указанные версии пакетов
  block:
    - name: Получить текущий список заблокированных пакетов
      ansible.builtin.command:
        cmd: dnf versionlock list
      register: versionlock_list
      changed_when: false
      check_mode: no

    - name: Для каждого пакета проверить и добавить блокировку
      ansible.builtin.include_tasks: lock_single_package.yml
      loop: "{{ packages_to_lock }}"
      loop_control:
        loop_var: pkg
```

#### `roles/lock_versions/tasks/lock_single_package.yml`

```yaml
---
- name: Определить, установлен ли пакет {{ pkg.name }} (опционально)
  ansible.builtin.command:
    cmd: rpm -q "{{ pkg.name }}"
  register: rpm_check
  failed_when: false
  changed_when: false

- name: Сформировать строку для блокировки (с версией или без)
  set_fact:
    lock_spec: "{{ pkg.name }}-{{ pkg.version }}"
  when: pkg.version is defined

- name: Если версия не указана, пытаемся взять текущую установленную
  set_fact:
    lock_spec: "{{ pkg.name }}-{{ rpm_check.stdout.split('-')[1:] | join('-') }}"
  when:
    - pkg.version is not defined
    - rpm_check.rc == 0

- name: Проверить, есть ли уже такая блокировка
  set_fact:
    already_locked: "{{ lock_spec in versionlock_list.stdout }}"

- name: Добавить блокировку через dnf versionlock add
  ansible.builtin.command:
    cmd: "dnf versionlock add {{ lock_spec }}"
  when: not already_locked
  register: lock_result

- name: Вывести результат добавления блокировки
  ansible.builtin.debug:
    msg: "Заблокирован пакет {{ lock_spec }} на хосте {{ inventory_hostname }}"
  when: lock_result is changed

- name: Логируем действие в общий лог
  ansible.builtin.lineinfile:
    path: "{{ playbook_dir }}/update.log"
    line: "{{ ansible_date_time.iso8601 }} - INFO - Хост {{ inventory_hostname }}: заблокирован пакет {{ lock_spec }}"
    create: yes
  delegate_to: localhost
  when: lock_result is changed
```

#### Альтернативный подход через exclude в dnf.conf (менее гибкий)

Если нужно не фиксировать конкретную версию, а просто исключить пакет из любых обновлений, можно использовать модуль `ini_file`:

```yaml
- name: Добавить exclude для git-lfs в /etc/dnf/dnf.conf
  ansible.builtin.ini_file:
    path: /etc/dnf/dnf.conf
    section: main
    option: exclude
    value: "git-lfs*"
    mode: '0644'
    backup: yes
  notify: restart dnf (не требуется, просто сброс кэша)
```

Но это глобальное исключение, не позволяющее зафиксировать конкретную версию.

### Объяснение для молодого коллеги (почему именно так)

#### Зачем фиксировать версии пакетов?

- **Совместимость**: ПО пользователя может тестироваться только с определённой версией `git-lfs`. Автоматическое обновление до мажорной версии может сломать скрипты, CI/CD пайплайны.
- **Контроль изменений**: в регламентированных средах (медицина, финансы) каждое обновление пакета должно быть согласовано. `versionlock` служит «страховкой» от случайного обновления.
- **Предсказуемость**: обновление системы (`dnf update`) не должно менять поведение пользовательских приложений. Локи дают гарантию, что критичные пакеты останутся на месте.

#### Почему ручной процесс через `dnf versionlock add`?

- Это стандартный механизм DNF, он записывает блокировки в файл `/etc/dnf/plugins/versionlock.list`. Плагин читается при каждом вызове `dnf update` или `dnf install`.
- В отличие от `exclude` в `dnf.conf`, `versionlock` позволяет зафиксировать **конкретную версию** (с точностью до сборки), а не всё имя пакета.
- Можно добавлять маски, например `git-lfs-3.2.*` – заблокировать все обновления в рамках минорной версии 3.2, но разрешить патчи.

#### Что делает наша Ansible-роль?

1. Устанавливает плагин `versionlock`, если его нет.
2. Для каждого пакета из списка `packages_to_lock`:
   - Проверяет, установлен ли пакет (через `rpm -q`). Если не установлен – можно либо пропустить, либо зафиксировать желаемую версию, чтобы при будущей установке не взялась более новая.
   - Формирует строку блокировки `имя-версия`.
   - Проверяет, нет ли уже такой строки в выводе `dnf versionlock list`.
   - Если нет – выполняет `dnf versionlock add`.
3. Логирует действие в общий лог-файл `update.log`.

**Почему это безопасно?**  
- Роль не перезаписывает существующие блокировки, а только добавляет отсутствующие.
- Она идемпотентна: повторный запуск не добавит дубликатов.
- Ошибки (например, неверная версия) приведут к провалу задачи, что остановит плейбук до обновления системы — это желаемое поведение.

#### Как это интегрируется с общим процессом обновления?

В `site.yml` роль `lock_versions` вызывается **ДО** всех проверок репозиториев и самого обновления. Почему?
- Если блокировка не была установлена, а пакет обновился бы автоматически, пользовательское ПО могло сломаться. Исправлять потом будет сложнее.
- Блокировка не мешает обновлению других пакетов и ядра.
- После успешного обновления блокировки остаются активными. Если позже понадобится обновить git-lfs, администратор явно удалит блокировку (руками или через Ansible-задачу).

#### Проверка работы после выполнения Ansible

На целевом хосте выполните:
```bash
sudo dnf versionlock list
```
Вы должны увидеть строку с `git-lfs` и зафиксированной версией.

Также можно проверить, что `dnf update --assumeno` не пытается обновить заблокированный пакет.

#### Обработка крайних случаев

- **Пакет не установлен**: в текущем коде блокировка добавится, даже если пакет отсутствует. Это полезно: если когда-нибудь пакет будет установлен, DNF возьмёт именно ту версию, которая зафиксирована. Если нужно блокировать только установленные пакеты — добавьте условие `when: rpm_check.rc == 0`.
- **Неверная версия**: `dnf versionlock add` завершится с ошибкой (неверный формат), задача упадёт, плейбук остановит выполнение для группы (благодаря `rescue` и `end_play`). Это правильно, т.к. конфигурация должна быть исправлена до обновления.
- **Плагин уже установлен**: задача `dnf name=python3-dnf-plugin-versionlock state=present` идемпотентна, повторно не устанавливает.

### Заключение

Добавление роли `lock_versions` повышает надёжность обновления в средах с жёсткими требованиями к совместимости. Она даёт возможность администратору декларативно описать, какие версии каких пакетов должны быть «заморожены», и автоматически применить это на всех хостах группы перед выполнением `dnf update`.
