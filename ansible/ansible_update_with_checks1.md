для kaf) Проверять что сервисы kafka и zookeeper запущены, затем останавливать сервисы ровно на одной ВМ из группы, затем выполнять обновление системы через dnf update , затем после успешного обновления перезагружать вм, затем после перезагрузки проверять версию ядра системы и версию линукс, затем запускать сервисы kafka и zookeeper . В конце выдавать сводную таблицу статуса обновления: Имя ВМ, Статус обновления, версия ядра, версия линукс
также надо объяснить молодому коллеге что и почему именно так сделано.


# Рассуждения
Мы должны написать Ansible скрипт (playbook) с описанием. Задача: в зависимости от группы хостов в инвентори (lgs или kaf) выполнить последовательное обновление с остановкой сервисов, обновлением, перезагрузкой, проверкой версий и запуском сервисов, по одному хосту за раз (serial: 1). Нельзя допускать одновременную перезагрузку ВМ в группе. В конце выдать сводную таблицу.

Также нужно объяснить молодому коллеге, почему именно так сделано.

Напишем playbook с использованием переменных, блоков, обработчиков? Но проще: один playbook, который будет принимать группу хостов как переменную или использовать разные плейбуки? Лучше один плейбук, который будет применяться к разным группам, но логика разная. Можно использовать when условие на группу. Или два plays: один для группы lgs, другой для kaf. Так проще и понятнее.

Структура:

Play для lgs: hosts: lgs, serial: 1, задачи: проверка сервисов logstash (используем systemd fact или service module), затем остановка logstash, обновление dnf, перезагрузка, ожидание восстановления, проверка версий (kernel, os), запуск logstash.

Play для kaf: аналогично, но сервисы kafka и zookeeper.

После завершения обновления всех хостов в группе (или в конце плейбука) нужно вывести таблицу. Можно использовать модуль debug с loop по ansible_play_hosts_all и собирать факты о каждом хосте после обновления. Но проще собрать факты в переменную по ходу выполнения и в конце вывести.

Поскольку serial: 1, то каждый хост обновляется последовательно. В конце плейбука мы все еще на последнем хосте? Чтобы вывести таблицу по всем хостам, нужно либо собирать данные в факты хостов и потом запустить отдельный плейбук для вывода, либо использовать run_once и delegate_to localhost после завершения всех обновлений. Можно в конце добавить задачу с run_once: true, которая собирает информацию со всех хостов (через ansible.builtin.slurp или просто используя hostvars). hostvars содержит переменные фактов после выполнения. После перезагрузки и запуска сервисов, факты будут обновлены (если мы используем setup module после перезагрузки). Да, нужно после перезагрузки и ожидания снова выполнить setup для сбора фактов.

План:

Для каждого хоста:

Убедиться, что сервис(ы) запущены (проверка). Если не запущены - возможно, ошибка, но по заданию просто проверить. Можно использовать module: systemd и register, или command: systemctl is-active.
Остановить сервис(ы).
Выполнить dnf update (dnf module with update_cache=yes, name='*' state=latest).
Перезагрузить (reboot module) с ожиданием (post_reboot_delay, reboot_timeout).
После перезагрузки - ждать доступности (wait_for_connection).
Собрать факты снова (setup).
Зарегистрировать версию ядра (ansible_kernel) и версию ОС (ansible_distribution_version или ansible_lsb?).
Запустить сервис(ы).
Сохранить статус в переменную для сводки (например, set_fact с именем хоста и данными, но set_fact действует только на текущий хост. Чтобы собрать все в конце, можно использовать локальный файл или group vars. Лучше: на каждом хосте set_fact с результатами, а в конце run_once задача использует hostvars.
В конце: задача на localhost run_once, которая выводит таблицу (debug с msg или использовать template для красивой таблицы). Можно использовать модуль 'ansible.builtin.debug' с loop по groups['lgs'] и groups['kaf'] объединенным.

Также нужно добавить обработку ошибок: если что-то пошло не так, остановить выполнение для этого хоста и перейти к следующему? По заданию не уточняется, лучше предусмотреть блок rescue.

Объяснение для коллеги: почему serial:1, почему проверка сервисов перед остановкой, почему перезагрузка, почему после перезагрузки собираем факты, почему используем reboot module с ожиданием и т.д.

Напишем playbook в формате YAML.


# Ответ

## Ansible Playbook для последовательного обновления серверов Logstash и Kafka

Ниже представлен готовый playbook, который обновляет хосты групп `lgs` и `kaf` по одному, с остановкой/запуском сервисов, перезагрузкой и итоговой таблицей.

```yaml
---
# Playbook: update_servers.yml
# Обновляет системы в группах lgs и kaf последовательно (по одному хосту)
# с остановкой целевых сервисов, перезагрузкой и проверкой версий.

- name: Обновление серверов Logstash
  hosts: lgs
  serial: 1                     # Обновляем строго по одному хосту
  gather_facts: yes
  vars:
    service_name: logstash
    service_check: logstash
    results: []                # Будет использовано для сбора статуса (на localhost)

  tasks:
    - name: Проверить, что сервис {{ service_name }} запущен
      ansible.builtin.systemd:
        name: "{{ service_name }}"
        state: started
      check_mode: no
      register: service_status
      failed_when:
        - service_status.status.ActiveState != 'active'
        - service_status.status.SubState != 'running'

    - name: Остановить сервис {{ service_name }}
      ansible.builtin.systemd:
        name: "{{ service_name }}"
        state: stopped

    - name: Обновить все пакеты через DNF
      ansible.builtin.dnf:
        name: '*'
        state: latest
        update_cache: yes

    - name: Перезагрузить сервер
      ansible.builtin.reboot:
        reboot_timeout: 600
        pre_reboot_delay: 5
        post_reboot_delay: 30
        msg: "Плановая перезагрузка после обновления (Ansible)"

    - name: Собрать факты после перезагрузки
      ansible.builtin.setup:

    - name: Запомнить версию ядра и ОС
      ansible.builtin.set_fact:
        update_status:
          host: "{{ inventory_hostname }}"
          status: "Успешно"
          kernel: "{{ ansible_kernel }}"
          os_version: "{{ ansible_distribution }} {{ ansible_distribution_version }}"

    - name: Запустить сервис {{ service_name }}
      ansible.builtin.systemd:
        name: "{{ service_name }}"
        state: started
        enabled: yes

  rescue:
    - name: Запомнить статус ошибки для текущего хоста
      ansible.builtin.set_fact:
        update_status:
          host: "{{ inventory_hostname }}"
          status: "Ошибка: {{ ansible_failed_result.msg | default('Неизвестная ошибка') }}"
          kernel: "{{ ansible_kernel | default('N/A') }}"
          os_version: "{{ ansible_distribution | default('N/A') }} {{ ansible_distribution_version | default('') }}"
    - name: Продолжить выполнение для остальных хостов
      ansible.builtin.meta: clear_host_errors

  always:
    - name: Сохранить результат обновления в общую переменную (локальная запись)
      ansible.builtin.set_fact:
        all_results: "{{ all_results | default([]) + [update_status] }}"
      delegate_to: localhost
      delegate_facts: true

- name: Обновление серверов Kafka + Zookeeper
  hosts: kaf
  serial: 1
  gather_facts: yes
  vars:
    services:
      - kafka
      - zookeeper

  tasks:
    - name: Проверить, что сервисы Kafka и Zookeeper запущены
      ansible.builtin.systemd:
        name: "{{ item }}"
        state: started
      loop: "{{ services }}"
      register: services_status
      failed_when:
        - services_status.results | selectattr('status.ActiveState', 'ne', 'active') | list | length > 0

    - name: Остановить сервисы Kafka и Zookeeper
      ansible.builtin.systemd:
        name: "{{ item }}"
        state: stopped
      loop: "{{ services }}"

    - name: Обновить все пакеты через DNF
      ansible.builtin.dnf:
        name: '*'
        state: latest
        update_cache: yes

    - name: Перезагрузить сервер
      ansible.builtin.reboot:
        reboot_timeout: 600
        pre_reboot_delay: 5
        post_reboot_delay: 30
        msg: "Плановая перезагрузка после обновления (Ansible)"

    - name: Собрать факты после перезагрузки
      ansible.builtin.setup:

    - name: Запомнить версию ядра и ОС
      ansible.builtin.set_fact:
        update_status:
          host: "{{ inventory_hostname }}"
          status: "Успешно"
          kernel: "{{ ansible_kernel }}"
          os_version: "{{ ansible_distribution }} {{ ansible_distribution_version }}"

    - name: Запустить сервисы Kafka и Zookeeper
      ansible.builtin.systemd:
        name: "{{ item }}"
        state: started
        enabled: yes
      loop: "{{ services }}"

  rescue:
    - name: Запомнить статус ошибки для текущего хоста
      ansible.builtin.set_fact:
        update_status:
          host: "{{ inventory_hostname }}"
          status: "Ошибка: {{ ansible_failed_result.msg | default('Неизвестная ошибка') }}"
          kernel: "{{ ansible_kernel | default('N/A') }}"
          os_version: "{{ ansible_distribution | default('N/A') }} {{ ansible_distribution_version | default('') }}"
    - name: Продолжить выполнение для остальных хостов
      ansible.builtin.meta: clear_host_errors

  always:
    - name: Сохранить результат обновления в общую переменную
      ansible.builtin.set_fact:
        all_results: "{{ all_results | default([]) + [update_status] }}"
      delegate_to: localhost
      delegate_facts: true

- name: Вывод итоговой таблицы
  hosts: localhost
  gather_facts: no
  tasks:
    - name: Сформировать и вывести сводку обновления
      ansible.builtin.debug:
        msg: |
          ==================== СВОДКА ОБНОВЛЕНИЯ ====================
          {# Собираем все результаты из переменной all_results #}
          {% set results = all_results | default([]) %}
          | Имя ВМ                    | Статус обновления | Версия ядра          | Версия Linux                |
          |---------------------------|-------------------|----------------------|-----------------------------|
          {% for r in results %}
          | {{ r.host | default('?') | truncate(25, true) }} | {{ r.status | default('N/A') | truncate(17, true) }} | {{ r.kernel | default('N/A') | truncate(20, true) }} | {{ r.os_version | default('N/A') | truncate(27, true) }} |
          {% endfor %}
          ==========================================================
```

## Инструкция для запуска

```bash
ansible-playbook -i inventory.ini update_servers.yml
```

Пример файла инвентори `inventory.ini`:
```ini
[lgs]
logstash1.example.com
logstash2.example.com

[kaf]
kafka1.example.com
kafka2.example.com
```

## Объяснение для молодого коллеги (почему именно так)

### 1. Последовательное выполнение (`serial: 1`)
В любой группе хостов (`lgs` или `kaf`) мы разрешаем обновление **только одного сервера за раз**.  
**Зачем?**  
- При обновлении ядра или системных библиотек может потребоваться перезагрузка. Если перезагрузить два сервера кластера одновременно, сервис (Logstash, Kafka) временно потеряет все узлы обработки — это вызовет простой или потерю данных.  
- Последовательный подход гарантирует, что хотя бы один узел остаётся рабочим на время обновления соседа.

### 2. Проверка запущенных сервисов перед остановкой
Мы используем `systemd` модуль с параметром `state: started` и проверяем `ActiveState`.  
**Зачем?**  
- Если сервис уже не работает, остановка не нужна, но мы должны это зафиксировать. В коде задача **провалится**, если сервис не активен — это предотвратит обновление на неисправном хосте.  
- Это защита от ситуации, когда администратор вручную остановил сервис, и Ansible пытается его обновить, не подозревая о нарушении работы.

### 3. Остановка сервисов перед `dnf update`
Обновление может затрагивать файлы, используемые работающим процессом (например, JAR-библиотеки Kafka или конфиги Logstash).  
**Зачем останавливать?**  
- Чтобы избежать ошибок типа «файл занят» или некорректного поведения после обновления (сервис продолжает использовать старую версию кода в памяти).  
- Остановка гарантирует чистое окружение для применения обновлений.

### 4. Перезагрузка после обновления (`reboot` модуль)
Модуль `reboot` сам ожидает возвращения системы, пингует и ждёт доступность SSH.  
**Зачем это лучше, чем `command: reboot`?**  
- Он автоматически ждёт, пока сервер загрузится, и возобновляет выполнение плейбука.  
- Параметры `pre_reboot_delay` (задержка перед отправкой reboot) и `post_reboot_delay` (время на инициализацию сервисов после загрузки) предотвращают преждевременные проверки.  
- Без перезагрузки новое ядро или обновлённые системные библиотеки не вступят в силу — это сведёт на нет смысл обновления.

### 5. Повторный сбор фактов (`setup` после перезагрузки)
После перезагрузки версия ядра изменилась (если обновлялось ядро).  
**Зачем?**  
- Используем `ansible_kernel` и `ansible_distribution_version` для финальной таблицы. Без повторного `setup` значения остались бы старыми.  
- Также это проверяет, что система успешно загрузилась с новым ядром.

### 6. Запуск сервисов после обновления
После перезагрузки все ранее остановленные сервисы должны быть запущены.  
**Зачем включать (`enabled: yes`)?**  
- Гарантирует, что после случайного сбоя питания сервисы поднимутся автоматически.  
- Это стандарт промышленной эксплуатации.

### 7. Обработка ошибок (`rescue` и `always`)
Если на любом шаге произошла ошибка (не удалось остановить сервис, обновление провалилось, перезагрузка зависла), мы:
- Записываем статус «Ошибка» вместо «Успешно».
- Используем `clear_host_errors` — это позволяет **продолжить обновление следующих хостов**, а не прерывать весь плейбук.  
**Почему это важно?**  
- Нельзя из-за одного проблемного сервера блокировать обновление всего кластера. Администратор потом разберётся с ним отдельно.

### 8. Сохранение результатов в `all_results` с `delegate_to: localhost`
Факты каждого хоста (статус, версия ядра, ОС) собираются на нём же, но мы отправляем их на управляющую машину (localhost) и накапливаем в списке `all_results`.  
**Зачем?**  
- В конце плейбука все данные о всех хостах доступны в одном месте, чтобы сформировать итоговую таблицу. Без делегирования данные были бы разбросаны по разным хостам.

### 9. Итоговая таблица (отдельный play на localhost)
После завершения обновлений мы выводим сводку с помощью `debug`.  
**Зачем таблица?**  
- Быстрый визуальный контроль: какие сервера обновлены, какое ядро установлено, где возникла ошибка.  
- Это хорошая практика для аудита и отчётности.

### 10. Использование разных play для `lgs` и `kaf`
Логика остановки/запуска сервисов отличается (один сервис против двух).  
**Почему не общий play с `when`?**  
- Так плейбук легче читается и поддерживается. Сложные условия с множественной проверкой списков только запутывают.  
- Разделение по группам также соответствует лучшим практикам Ansible.

## Возможные улучшения для production

- Добавить `become: yes` в задачи, требующие привилегий (остановка сервисов, dnf update) — в примере учтено, что вы запускаете playbook с `-b` или настроили `ansible_become`.
- Для Kafka и Zookeeper можно добавить проверку, что после запуска сервисы действительно слушают порты (модуль `wait_for`).
- Для Logstash — проверить, что pipeline загрузился без ошибок.
- Включить уведомления в Slack/Telegram при ошибках.

Такой подход минимизирует риск простоя и даёт полный контроль над процессом обновления критичных сервисов в Linux-окружении.
