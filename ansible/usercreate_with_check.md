# Вопрос

Ты администратор с 20 летним опытом работы. Твоя задача написать ansible playbook  который проверит наличие каталога (имя каталога cheking_path задается через переменные), затем если каталог не существует, то создаст пользователя (имя пользователя user_nane задается через переменные) и разместит публичный ключ пользователя (берем из переменной pub_key) в файле /home/<username>/.ssh/authorized_keys Plabook планируется исполнять на всех VM  из списка в файле hosts.ini

# Рассуждения

Мы должны написать Ansible playbook, который:

Проверяет наличие каталога (путь задается переменной checking_path).

Если каталог не существует, то создает пользователя (имя задается переменной user_name) и размещает публичный ключ (переменная pub_key) в /home/<username>/.ssh/authorized_keys.

Playbook выполняется на всех VM из списка в файле hosts.ini.

Дополнительные детали:

Пользователь должен быть создан только если каталог отсутствует.

Публичный ключ размещается только если каталог отсутствует (то есть условие одно и то же).

Возможно, нужно также создать домашнюю директорию пользователя, если она не создается автоматически.

Необходимо учитывать, что файл authorized_keys должен иметь правильные права (обычно 600) и директория .ssh (700).

Также в задании сказано "проверит наличие каталога (имя каталога cheking_path задается через переменные)" - возможно, опечатка "cheking_path" -> "checking_path". Будем использовать checking_path.

Файл hosts.ini - это инвентарный файл. Мы можем указать в playbook, что используем его.

Напишем playbook с использованием модулей:

stat: для проверки существования каталога.

user: для создания пользователя (с опцией create_home: yes, shell: /bin/bash и т.д.)

authorized_key: для добавления публичного ключа (или просто копирования содержимого, но authorized_key удобнее).

Возможно, потребуется создать директорию .ssh и установить права, но authorized_key сам создает директорию, если нужно, и устанавливает права.

Однако условие: "разместит публичный ключ пользователя (берем из переменной pub_key) в файле /home/<username>/.ssh/authorized_keys". Можно использовать модуль copy или lineinfile, но authorized_key предпочтительнее.

Учитывая, что пользователь создается только если каталог не существует, нужно объединить эти задачи в блок с условием when: not checking_path_exists.stat.exists.

Но сначала нужно зарегистрировать результат stat.

Также нужно определить переменные: checking_path, user_name, pub_key. Они могут быть заданы в playbook как vars, или переданы через extra-vars, или в инвентаре. Поскольку в задании сказано "задается через переменные", мы можем определить их в playbook или ожидать, что они будут предоставлены. Для примера, зададим их в vars секции playbook, но с возможностью переопределения.

Проверим, что каталог существует и является директорией.

Также нужно учесть, что создание пользователя может потребовать sudo прав, поэтому playbook должен выполняться с become: yes.

Напишем playbook.
~~~
Структура:

name: Check directory and create user if missing
hosts: all
gather_facts: yes # чтобы узнать, например, домашнюю директорию, но мы используем фиксированную /home/
vars:
checking_path: /some/path # пример, можно переопределить
user_name: myuser
pub_key: "ssh-rsa AAAAB3NzaC1yc2EAAAADAQABAAABAQ..." # пример ключа
tasks:

name: Check if directory exists
stat:
path: "{{ checking_path }}"
register: dir_stat

name: Create user and add SSH key if directory does not exist
block:

name: Create user
user:
name: "{{ user_name }}"
state: present
create_home: yes
shell: /bin/bash

можно указать home: "/home/{{ user_name }}" (по умолчанию так и есть)
name: Add SSH public key for user
authorized_key:
user: "{{ user_name }}"
key: "{{ pub_key }}"
state: present

модуль authorized_key сам создаст .ssh и установит права
when: not dir_stat.stat.exists or not dir_stat.stat.isdir

лучше проверить, что директории нет или это не директория (на случай, если это файл)
используем условие: not (dir_stat.stat.exists and dir_stat.stat.isdir)
~~~
Но условие "если каталог не существует" означает, что его нет. В случае, если есть файл с таким именем, то это тоже "не существует как каталог". Мы можем проверять: не dir_stat.stat.exists или (dir_stat.stat.exists and not dir_stat.stat.isdir). Упростим: not (dir_stat.stat.exists and dir_stat.stat.isdir). Так мы создадим пользователя, если каталога нет или вместо каталога что-то другое.

Однако, возможно, логика требует именно проверки на существование каталога, а если есть файл, то это тоже "не каталог", значит условие сработает. Но возможно, это не ожидаемое поведение. Для простоты будем считать, что проверяем именно существование каталога: when: not dir_stat.stat.isdir.

Однако, если пути нет, stat вернет exists: false, isdir: false. Условие not dir_stat.stat.isdir будет true (так как false), и мы создадим пользователя. Если есть файл, isdir: false, тоже true. Если есть каталог, isdir: true, условие false.

Поэтому используем when: not dir_stat.stat.isdir.

Дополнительно: после создания пользователя, возможно, нужно создать сам каталог? В задании не сказано, только проверить наличие каталога и если нет - создать пользователя. Каталог, возможно, создается другой задачей, не в этом playbook. Поэтому мы не создаем каталог.

Также нужно учесть, что пользователь может уже существовать. Модуль user создаст его только если нет, так что это безопасно.

Теперь о выполнении на всех VM из hosts.ini. Это просто указание hosts: all и использование инвентарного файла hosts.ini при запуске: ansible-playbook -i hosts.ini playbook.yml.

В самом playbook мы не указываем инвентарь, это внешнее.

Также можно задать переменные в файле group_vars/all.yml или передавать через командную строку, но для примера оставим в vars.

Напишем ответ: нужно предоставить playbook с объяснением.

Поскольку задание: "Ты администратор с 20 летним опытом работы. Твоя задача написать ansible playbook который ...". Ответ должен быть в виде готового playbook с комментариями.

Также уточним: "разместит публичный ключ пользователя (берем из переменной pub_key) в файле /home/<username>/.ssh/authorized_keys". Мы используем модуль authorized_key.

Теперь напишем итоговый playbook.

# Ответ

Ниже представлен Ansible playbook, который выполняет указанные задачи. Playbook использует переменные для гибкости и включает проверки для корректной обработки условий.

```yaml
---
- name: Проверка каталога и создание пользователя при его отсутствии
  hosts: all              # Выполняется на всех хостах из inventory (hosts.ini)
  become: yes             # Требуются привилегии root для создания пользователя
  gather_facts: no        # Отключаем сбор фактов для ускорения (при необходимости можно включить)

  vars:
    # Переменные можно переопределить при запуске через --extra-vars
    checking_path: /opt/required_dir   # Путь для проверки
    user_name: ansible_user            # Имя создаваемого пользователя
    pub_key: "ssh-rsa AAAAB3NzaC1yc2EAAAADAQABAAABAQ..."  # Публичный ключ пользователя

  tasks:
    - name: Проверить существование каталога
      ansible.builtin.stat:
        path: "{{ checking_path }}"
      register: dir_check

    - name: Создать пользователя и добавить SSH-ключ, если каталог отсутствует
      block:
        - name: Создать системного пользователя
          ansible.builtin.user:
            name: "{{ user_name }}"
            state: present
            create_home: yes           # Автоматически создаст /home/<user_name>
            shell: /bin/bash           # Устанавливаем оболочку по умолчанию
            # При необходимости можно добавить группу, uid и т.д.

        - name: Добавить публичный ключ в authorized_keys
          ansible.posix.authorized_key:
            user: "{{ user_name }}"
            key: "{{ pub_key }}"
            state: present
            # Модуль сам создаст каталог .ssh с правами 700 и файл authorized_keys с правами 600
      when: not dir_check.stat.isdir   # Условие: каталог не существует или это не каталог
```

### Пояснения:
1. **Проверка каталога**  
   Модуль `stat` проверяет наличие пути `{{ checking_path }}`. Результат сохраняется в переменную `dir_check`.

2. **Условное выполнение блока**  
   Блок задач выполняется только если `dir_check.stat.isdir` равен `false` (каталог не существует или путь указывает на файл).  
   *Важно:* Если путь существует, но является файлом (не каталогом), условие также сработает, и пользователь будет создан.

3. **Создание пользователя**  
   Модуль `user` создаёт пользователя с домашней директорией `/home/{{ user_name }}` и оболочкой `/bin/bash`. Если пользователь уже существует, модуль ничего не изменит.

4. **Добавление SSH-ключа**  
   Модуль `authorized_key` (из коллекции `ansible.posix`) добавляет ключ в файл `~{{ user_name }}/.ssh/authorized_keys`. При необходимости он самостоятельно создаёт каталог `.ssh` (с правами 700) и файл (с правами 600), что соответствует стандартным требованиям безопасности.

### Использование:
- Сохраните playbook, например, как `create_user_if_dir_missing.yml`.
- Подготовьте файл инвентаря `hosts.ini` с перечнем целевых VM:
  ```ini
  [all]
  vm1 ansible_host=192.168.1.10
  vm2 ansible_host=192.168.1.11
  ```
- Запустите playbook, передав при необходимости свои значения переменных:
  ```bash
  ansible-playbook -i hosts.ini create_user_if_dir_missing.yml \
    -e "checking_path=/opt/data user_name=deployer pub_key='ssh-rsa AAA...'"
  ```

### Примечания:
- Если каталог уже существует, задачи создания пользователя и ключа будут пропущены.
- Убедитесь, что на целевых хостах установлен Python и Ansible имеет права на выполнение с `become`.
- Для работы модуля `authorized_key` требуется коллекция `ansible.posix`. Установите её при необходимости:  
  `ansible-galaxy collection install ansible.posix`
